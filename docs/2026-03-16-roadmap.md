> 작성일: 2026-03-16
> 기준: 전체 코드베이스 + 문서 종합 검토

---

## 판단 기준

각 항목은 아래 4개 축으로 평가했다.

| 축 | 의미 |
|----|------|
| **제품 가치** | 실제 사용자 업무에 얼마나 직결되는가 |
| **기술 부채** | 지금 안 하면 나중에 더 비싸지는가 |
| **의존 관계** | 이것이 막혀야 다른 것도 막히는가 |
| **난이도** | 실제 구현 공수 |

## 현재 상태 요약

```
완성된 것
├── 내부 MCP 서버 (aimd-local) — tools/resources/prompts 완전 동작
├── 전체 UI 디자인 시스템 준수 (glassmorphism 제거, CSS 변수 전환 완료)
├── Supabase 프로젝트 연결 + 마이그레이션 적용 (pwllacujwgzulkelqfrq)
├── 핵심 워크플로우 (스튜디오 → 세미나 → 보고서 → 승인 → 플레이북)
└── Electron 패키징 + 자동 업데이트 (v0.2.0)

막혀 있는 것
├── Supabase Auth 이메일 발송 제한 (email rate limit exceeded)
└── 외부 MCP 8개 모두 레지스트리만 있고 실제 미연결

남은 기술 부채
├── status color 하드코딩 (6개 페이지)
├── app-nav hex 색상 4곳
└── seminar 라운드 상태 hex 색상

SQLite → Supabase 이전
└── 마이그레이션 설계 완료, 실제 데이터 이전 미시작
```

## 🔴 Phase 1 — 이번 주 (즉시 실행)

### 1-A. Supabase 이메일 차단 해제
**왜 지금**: Auth가 막혀 있으면 Supabase 전환 전체가 블로킹됨

- Supabase 대시보드 → Auth → SMTP 설정 → 외부 SMTP 연결 (Resend / SendGrid / AWS SES 중 택1)
- Resend 권장 (무료 3,000건/월, 설정 5분)
- 완료 후: 이메일 로그인 → 세션 연결 → 워크스페이스 생성 흐름 전체 확인

---

### 1-B. MCP 레지스트리 확장 (4개 신규 추가)
**왜 지금**: 파일 1개 수정, 30분, 이후 모든 MCP 작업의 기반

`lib/mcp-connections.ts`에 추가:

| ID | 패키지 | Wave |
|----|--------|------|
| `supabase` | `@supabase/mcp-server-supabase` | 1 |
| `slack` | `@modelcontextprotocol/server-slack` | 2 |
| `gdrive` | `@modelcontextprotocol/server-gdrive` | 2 |
| `brave-search` | `@modelcontextprotocol/server-brave-search` | 3 |

상세 스키마 → `docs/2026-03-16-mcp-connection-plan.md` 참조

---

### 1-C. Playwright MCP 실제 연결
**왜 지금**: 이미 `/api/mcp/playwright/smoke` 엔드포인트와 UI 준비됨. 설정만 하면 동작

- 설정 화면 → 개발 점검 모드 → Playwright MCP → command 등록
- `npx @playwright/mcp@latest` 명령어 입력 후 활성화
- smoke test 4개(홈, 설정, 데이터, 세미나) 실행 확인

---

### 1-D. 디자인 시스템 status badge 토큰화
**왜 지금**: 6개 페이지에서 동일 코드 반복 중. 이후 화면 추가할 때마다 복사 오류 위험

```
globals.css        → .status-badge-* 클래스 추가
lib/design-tokens.ts → getStatusColor() 유틸리티 생성
```

영향 파일:
- `app/operations/page.tsx` — `seminarTone()`, `timelineTone()`
- `app/campaigns/page.tsx` — `statusTone()`
- `app/campaigns/[id]/page.tsx` — `roomStatusTone()`, `timelineTone()`, `coverageTone()`
- `app/seminar/page.tsx` — `statusClass`, 라운드 상태 hex
- `app/learning/page.tsx` — `statusTone()`
- `app/dashboard/page.tsx` — `statusTone()`

---

### 1-E. app-nav.tsx + seminar hex 색상 정리
**왜 지금**: 디자인 시스템 마무리. 30분

```tsx
// app-nav.tsx — hex → CSS 변수
'bg-[rgba(49,130,246,0.1)] text-[#3182f6]'  →  'bg-[var(--accent-soft)] text-[var(--accent)]'
'text-[#6b7684] hover:bg-[#f5f6f7]'          →  'text-[var(--text-muted)] hover:bg-[var(--surface-sub)]'
'bg-[#e8ebed]'                                →  'bg-[var(--surface-border)]'

// app/seminar/page.tsx — 라운드 상태 hex 제거
'#e7f7ee', '#304f7a', '#8a3636' → status badge 클래스 적용
```

## 🟠 Phase 2 — 다음 주 (기능 확장)

### 2-A. Notion MCP 실제 연결
**제품 가치 최상**: 실행 결과물이 팀과 공유되는 첫 번째 외부 연결

구현 순서:
1. Notion Integration 생성 → API Key 발급
2. 설정 화면 → Notion 연결 패널에 OAuth/API Key 입력 플로우 추가
3. 연결 후 대상 DB/페이지 선택 UI
4. 화면별 발행 버튼 추가:
   - `app/seminar/page.tsx` → "보고서 Notion 발행" 버튼
   - `app/learning/page.tsx` → "플레이북 Notion 저장" 버튼
   - `app/operations/page.tsx` → "오늘의 브리핑 공유" 버튼

---

### 2-B. Supabase MCP 연결
**제품 가치 높음**: 이미 연결된 DB를 Claude가 직접 읽고 분석

- Supabase 대시보드 → Account → Access Tokens → 발급
- 설정 화면 → Supabase MCP → bearer token 입력 후 활성화
- 연결 확인: `workspace_runs`, `workspace_learning_archives` 쿼리 테스트

---

### 2-C. Supabase 데이터 이전 Phase 1-2
**왜 지금**: Auth가 열리면 바로 진행해야 함. 팀 협업의 기반

**Phase 1 - Auth + 조직 모델** (현재 마이그레이션 이미 있음)
- `profiles`, `organizations`, `organization_memberships` 정상 동작 확인
- 워크스페이스 생성 → 멤버 초대 흐름 테스트

**Phase 2 - 핵심 운영 데이터 이전**
이전 대상 (우선순위 순):
```
1순위  Run, MeetingTurn, Deliverable, MemoryLog
2순위  LearningArchive, ApprovalDecision
3순위  SeminarSession, SeminarRound, SeminarFinalReport
4순위  RunProgress (Realtime 연동)
```

각 테이블에 `organization_id`, `created_by` 컬럼 추가 후 RLS 적용

---

### 2-D. Supabase Storage 분리
**왜 지금**: 현재 `RunAttachment.content`와 `Dataset.rawData`가 DB에 blob으로 들어가 있어 DB 비대 문제

bucket 설계:
```
run-attachments/  — 첨부 파일 원본
datasets/         — CSV/XLSX 원본
reports/          — PDF 보고서
```

- `file_assets` 메타 테이블 생성
- 업로드 시: 원본 → Storage, 텍스트 추출 요약 → DB 메타 필드

---

## 🟡 Phase 3 — 2-3주 후 (연결 확장)

### 3-A. Slack MCP 연결 + 승인 알림 워크플로우
**제품 가치**: 승인 대기가 팀 채널로 자동 알림 → 병목 해소

구현:
- Slack App 생성 → Bot Token 발급
- 설정 화면 → Slack MCP 연결
- `app/campaigns/page.tsx` → "팀에 공유" 버튼 (승인 대기 항목)
- `app/operations/page.tsx` → "오늘의 브리핑 채널 발송" 버튼

---

### 3-B. Supabase Realtime 적용
**제품 가치**: 승인/진행 상태가 새로고침 없이 반영

대상 3개만:
```
RunProgress         — 실행 진행률 실시간
ApprovalDecision    — 승인 상태 실시간
SeminarSession      — 세미나 라운드 상태 실시간
```

---

### 3-C. Instagram Login 전환
**왜 지금**: 현재 Meta Business 연결은 사내 1개 계정 목적에 오버스펙. Instagram Login이 더 단순

- `components/meta-connection-panel.tsx` → Instagram Login API 우선 흐름으로 재설계
- Meta Business 연결은 고급 옵션으로 후퇴

---

### 3-D. Google Drive MCP 연결
**제품 가치**: 데이터셋 수동 업로드 → Drive 폴더 자동 동기화

- `app/datasets/page.tsx` → "Drive에서 가져오기" 버튼
- OAuth 플로우 설정 화면에서 처리
- 업로드 스튜디오와 Drive 연결

---

### 3-E. 내부 MCP 서버 확장
**왜 지금**: 현재 6개 tools로는 외부 분석 요청 커버 부족

추가할 tools:
```typescript
get_seminar_session_detail  // 세미나 세션 전체 상세
list_approval_queue         // 현재 승인 대기 항목
get_campaign_room_detail    // 캠페인 룸 상세
get_kpi_goals_summary       // KPI 목표 현황
search_learning_by_signal   // 신호 태그 기반 학습 카드 검색
```

추가할 prompts:
```typescript
campaign-room-briefing      // 캠페인 룸 기준 오늘의 브리핑 생성
approval-action-summary     // 승인 대기 항목 분석 + 처리 권고
kpi-progress-review         // KPI 진행 상황 주간 리뷰
```

## 🟢 Phase 4 — 1개월+ (기능 심화)

### 4-A. 추천 액션 엔진
캠페인 상태, 승인 대기, KPI 달성률을 기반으로 "지금 할 일" 자동 추천
- `app/operations/page.tsx` → 추천 카드 섹션 강화
- 내부 MCP의 `aimd://overview` 리소스 확장으로 구현 가능

### 4-B. 세미나 전략 시뮬레이터 강화
- 라운드별 맥락 누적 개선
- 외부 데이터(인스타그램, 데이터셋)를 세미나 컨텍스트로 자동 주입
- 세미나 → 플레이북 전환율 트래킹

### 4-C. 캠페인 데이터 모델 고도화
현재 `ManualCampaignRoom`이 너무 단순. 추가 필요:
- `campaign_kpis`, `campaign_channels`, `campaign_actions`
- 캠페인 단위 성과 집계

### 4-D. Sentry + GitHub + Vercel MCP
개발/운영 자동화. 제품 기능보다 팀 운영 효율 목적

### 4-E. Brave Search MCP
캠페인 스튜디오에서 실시간 시장 조사 → 현재 웹 검색 기능 대체/보완
