## 1. 한 줄 정의

Garnet은 더 이상 `AI 마케팅 운영 플랫폼`이 아니라,
사용자의 업무 방식과 판단 기준을 축적하고 진화시키는 `개인 에이전트 시스템`이다.

마케팅은 Garnet이 처음으로 깊게 지원하는 전문 도메인이며,
장기적으로는 여러 업무 영역으로 확장 가능한 첫 번째 vertical 이다.

## 2. 왜 피벗해야 하는가

현재 Garnet은 이미 다음 요소를 갖고 있다.

- 역할 기반 실행 구조
- 승인 흐름
- 학습 자산화
- 자동화 스케줄러
- MCP를 통한 외부 도구 연결
- 장기적으로 반복 작업을 축적할 수 있는 저장 레이어

하지만 지금의 제품 정의는 여전히 `마케팅 앱`에 머물러 있다.
이 상태에서는 기능이 늘수록 제품 중심이 `캠페인`, `브리핑`, `세미나`, `SNS 운영`에 고정된다.

반대로 두 기준 문서가 제안하는 방향은 다음과 같다.

- 개인 자산으로 축적되는 Core
- 외부 의존 요소와 분리 가능한 구조
- 자기개선 루프를 가진 시스템
- 승인 기반 자율성
- 여러 하위 에이전트로 확장 가능한 구조

즉 Garnet의 다음 단계는 `마케팅 기능 강화`가 아니라
`개인 에이전트 Core를 세우고, 마케팅을 첫 번째 도메인 모듈로 재배치하는 것`이다.

## 3. 제품 비전

### 비전

사용자가 매번 같은 방식으로 생각하고, 정리하고, 조사하고, 승인하고, 반복하지 않아도
Garnet이 그 패턴을 학습하고 재사용해 점점 더 나를 닮아가는 개인 에이전트가 된다.

### 제품 약속

Garnet은 다음을 제공해야 한다.

- 내가 자주 하는 일을 기억한다
- 내가 승인하는 기준을 학습한다
- 내가 쓰는 도구들을 연결해준다
- 내가 직접 통제할 수 있는 자율성만 허용한다
- 외부 서비스가 바뀌어도 내 Core 자산은 남는다

## 4. 제품 원칙

두 기준 문서를 바탕으로 Garnet의 상위 원칙을 다음처럼 정리한다.

### 4.1 Personal Asset First

Garnet의 핵심 가치는 기능 수가 아니라
사용자 고유의 메모리, 선호, 워크플로우, 승인 패턴이 자산으로 남는 데 있다.

### 4.2 Core vs Domain Separation

다음은 Core 에 속한다.

- 메모리
- 승인 정책
- 작업 실행 구조
- 자기개선 루프
- 도구 연결
- 사용자 선호와 반복 패턴

다음은 Domain 에 속한다.

- 마케팅
- SNS 운영
- 데이터 분석 템플릿
- 특정 외부 API 연동 로직

### 4.3 Approval-Based Autonomy

자동화의 목표는 완전 무인화가 아니라
사용자가 신뢰할 수 있는 수준까지 점진적으로 자율성을 높이는 것이다.

### 4.4 Self-Improvement with Guardrails

Garnet은 스스로 구조를 관찰하고, 개선안을 제안하고, 초안을 만들 수 있어야 한다.
단, 고위험 변경은 항상 승인과 검증을 통과해야 한다.

### 4.5 Portable Core

모델, 공급자, 인프라, 도구가 바뀌어도
Garnet의 Core 구조와 사용자 자산은 유지 가능해야 한다.

## 5. 피벗 후 제품 정의

### 이전 정의

- AI 마케팅 OS
- 캠페인 중심 운영 도구
- 마케팅 분석과 실행 자동화가 중심

### 이후 정의

- 개인 에이전트 시스템
- 메모리, 승인, 계획, 실행, 자기개선이 중심
- 마케팅은 첫 번째 전문 모듈

### 피벗 후 설명 문구 초안

`Garnet은 사용자의 업무 방식, 판단 기준, 반복 작업을 학습하고 실행하는 개인 에이전트 시스템입니다. 현재는 마케팅 도메인을 가장 깊게 지원하며, 점차 여러 업무 영역으로 확장됩니다.`

## 6. 무엇을 유지하고 무엇을 바꿀 것인가

### 유지할 것

- 현재 스케줄러와 잡 실행 인프라
- 승인 인박스와 승인 기록
- 학습 카드/메모리 저장 흐름
- MCP 연결 허브
- Tauri + Next.js 기반 로컬 우선 구조

### 재배치할 것

- `캠페인 스튜디오`, `캠페인 룸`, `세미나`, `마케팅 인텔` 등은 `domain-marketing` 아래로 이동
- 현재 마케팅 전용 프롬프트는 Core prompt 가 아니라 도메인 prompt 로 분리
- 제품 홈은 마케팅 홈이 아니라 에이전트 홈으로 교체

### 보류할 것

- 마케팅 세부 기능의 추가 확장
- SNS/광고 고도화 중심 로드맵
- 자동 브랜치/PR 생성
- 낮은 위험 작업 자동 반영

## 7. 피벗 후 핵심 사용자 경험

개인 에이전트로서 Garnet의 핵심 흐름은 다음이어야 한다.

1. 사용자가 해야 할 일, 생각 중인 일, 자료를 넣는다
2. Garnet이 맥락을 기억하고 작업 유형을 파악한다
3. 필요하면 조사, 정리, 비교, 초안 작성, 실행 계획을 만든다
4. 중요한 변경이나 실행은 승인 대기함에 올린다
5. 결과와 승인 패턴을 학습해 다음 제안을 더 잘한다

### 초기 UX 축

- Today: 지금 중요한 일, 대기 승인, 추천 액션
- Inbox: 제안, 승인 요청, 회고 항목
- Memory: 학습 패턴, 선호, 반복 템플릿
- Workstreams: 프로젝트/업무 단위 진행 관리
- Research: 조사 결과와 외부 패턴 수집
- Domains: 마케팅 등 전문 모듈 진입

## 8. 마케팅 도메인의 새 위치

마케팅 기능은 더 이상 제품 정의 그 자체가 아니다.

대신 다음과 같이 위치를 바꾼다.

- Garnet Core
  - 메모리
  - 승인
  - 계획/실행
  - 자기개선
  - 도구 연결
- Domain: Marketing
  - Campaign workflows
  - Content generation
  - GA4 / SNS / Intel
  - Seminar / deliberation

이렇게 하면 마케팅은 버려지지 않고,
오히려 Garnet이 실제로 잘하는 첫 번째 전문 업무 영역으로 남는다.

## 9. 제품 로드맵 재구성

### Phase 0. Pivot Declaration

목표:

- 제품 정의 변경
- 기존 문서 기준선 정리
- 마케팅 중심 언어를 Core/Domain 관점으로 재정렬

주요 산출물:

- 새 제품 정의 문구
- 피벗 기획안
- 기술 설계 문서
- 기준 문서 업데이트

### Phase 1. Core Extraction

목표:

- 현재 코드에서 Core 후보와 domain-marketing 분리

주요 산출물:

- Core module map
- Domain module map
- 추상화 대상 개념 정의

### Phase 2. Generic Memory and Approval

목표:

- 마케팅 전용이 아닌 범용 메모리와 승인 구조 확립

주요 산출물:

- UserPreferenceMemory
- WorkflowTemplate
- ApprovalInbox 확장
- ChangeJournal 초안

### Phase 3. Self-Improvement Level 0-1

목표:

- 관찰 -> 분석 -> 제안 -> 승인 요청 루프 연결

주요 산출물:

- ImprovementProposal
- RiskAssessment
- ResearchTask
- 제안 인박스

### Phase 4. Domain Expansion Readiness

목표:

- 마케팅 외 다른 도메인도 수용 가능한 구조 확보

후보:

- 개인 생산성
- 문서/회의 운영
- 리서치 에이전트
- 개발 보조 워크플로우

## 10. 성공 지표

피벗 이후 성공을 다음 지표로 본다.

### 제품 지표

- 마케팅 외 범용 작업 처리 비율
- 반복 작업 재사용률
- 승인 요청 대비 승인율
- 제안 수용률
- Memory 재활용 빈도

### 시스템 지표

- 자율 제안 실패율
- 회귀 검증 실패율
- 고위험 작업 자동 차단율
- 사용자 선호 기반 응답 일관성

### 장기 지표

- 사용자가 직접 재설정하지 않고도 유지되는 개인화 정도
- 신규 도메인 추가 시 Core 수정량 감소

## 11. 리스크

### 11.1 정체성 혼선

마케팅 앱과 개인 에이전트가 동시에 전면에 있으면 제품 메시지가 흐려진다.

대응:

- Core와 domain-marketing의 위계를 문서와 IA에서 명확히 반영

### 11.2 과도한 범용화

너무 빨리 일반화하면 현재 잘 작동하는 기능까지 망가질 수 있다.

대응:

- 내부 구현은 단계적 추상화
- 기존 마케팅 워크플로우는 adapter 방식으로 유지

### 11.3 자기개선 자동화 과속

자동 코드 생성/반영을 너무 빨리 열면 품질과 신뢰를 잃는다.

대응:

- Level 0-1만 먼저 허용
- 회귀 검증과 리스크 평가 이후에만 Level 2+

## 12. 당장 실행할 우선순위

### 이번 단계

1. 제품 설명과 핵심 문서에서 `AI 마케팅 OS` 정체성을 걷어낸다
2. 현재 기능을 `Core` 와 `domain-marketing` 으로 분류한다
3. 범용 추상화 대상 모델을 결정한다
4. 자기개선용 제안 모델과 승인 흐름 초안을 설계한다

### 다음 단계

1. 에이전트 홈/인박스 중심 IA 초안
2. 범용 메모리 구조와 사용자 선호 저장 도입
3. ResearchTask -> ImprovementProposal 루프 구현

## 13. 최종 판단

두 기준 문서는 Garnet의 방향을 바꾸기에 충분하다.
중요한 것은 이 문서들을 `새 기능 아이디어`로 소비하지 않는 것이다.

이 문서들은 Garnet에 다음을 요구한다.

- 제품 정의를 바꿀 것
- Core를 세울 것
- 마케팅을 첫 번째 도메인으로 재배치할 것
- 자기개선 루프를 승인 기반으로 설계할 것

즉, Garnet의 다음 단계는 `더 큰 마케팅 플랫폼`이 아니라
`개인 에이전트 Core 위에 마케팅 전문성을 올린 시스템`이다.
